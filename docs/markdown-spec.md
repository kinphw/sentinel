# 개조식 Markdown 입력 사양 (Sentinel ↔ Sentinel-Forge 계약)

> ⚠️ **이 파일은 Sentinel과 Sentinel-Forge 두 프로젝트에 동시에 존재합니다.**
>
> - `c:/projects/sentinel-forge/spec/markdown-spec.md` (Forge 측 — STAGE 1 파서가 받아들이는 사양)
> - `c:/projects/sentinel/docs/markdown-spec.md` (Sentinel 측 — STAGE 2가 산출해야 하는 사양)
>
> 양 파일은 **항상 동일한 내용**을 유지해야 합니다.
> 변경 시 두 위치 동시 갱신 + § 8 변경 이력 기록 + 양 프로젝트 합의.

본 사양은 Sentinel STAGE 2가 산출하고 Sentinel-Forge가 입력으로 받는 개조식 markdown 형식을 정의한다.

**핵심 원칙 — 의미만, 서식은 Forge에**
- 마크다운은 **논리 구조**(층위·요약단어·주석·강조·결론·callout)만 표기한다.
- 시각적 서식(폰트·크기·자간·색상·들여쓰기·표 스타일·페이지 레이아웃 등)은 **모두 Sentinel-Forge가 책임**진다.
- 본 사양은 Forge가 식별해야 할 **7종 의미**(메타데이터·층위·요약단어·주석·강조·결론 화살표·callout 박스)만 정의한다.

**버전**: 1.3
**최종 갱신일**: 2026-04-26

---

## 1. 문서 골격

```
[메타데이터]    ← 문서 가장 처음, YAML front-matter
[본문]          ← 5단계 개조식
```

본문 분량 초과 시 [붙임] 분리는 Forge가 자동 처리. 마크다운에서 별도 표기하지 않는다.

---

## 2. 메타데이터

문서 최상단에 YAML front-matter 형식으로 3개 항목을 명시한다.

```yaml
---
보고서명: 금융결제원 뱅크페이서비스의 PG업 등록 필요여부 검토
작성부서: 전자금융감독국
작성일: 2026-04-25
---
```

- 키 이름은 정확히 위 3개(`보고서명`, `작성부서`, `작성일`).
- `작성일`은 `YYYY-MM-DD` 형식.
- 메타데이터의 시각적 표현(가운데 정렬·폰트·위치 등)은 Forge가 결정.

## 3. 본문 층위 (6단계)

| 레벨 | 글머리 | 용도 |
|:-:|:-:|---|
| 1 | `1.` `2.` `3.` ... | 섹션 헤더 (예: `1. 현황`) |
| 2 | `가.` `나.` `다.` ... `하.` | **소제목** — 섹션 내 하위 분류 (예: `가. 개요`) |
| 3 | `□` | 1차 분기 — 요약단어 필수 (§ 4) |
| 4 | `○` | 2차 분기 |
| 5 | `-` | 3차 분기 |
| 6 | `·` | 4차 분기 (최하위) |

### 식별 규칙

- 레벨은 **라인 시작의 글머리 문자**만으로 식별한다.
- 글머리 문자(`1.`/`가.`/`□`/`○`/`-`/`·`)와 주석·결론·callout 마커(`*`/`※`/`=>`/`[참고]`/`[붙임]`)는 모두 라인 시작 첫 문자(또는 첫 토큰)가 서로 충돌하지 않는다.
- 소제목 글머리는 **한글 가나다 음절 + 마침표 + 공백 + 본문**: `가.` `나.` `다.` ... `하.` (14개). 보통 한 섹션 내 1~5개 사용.
- **들여쓰기는 작성자의 가독성용으로 자유롭게 사용**한다. 파서는 들여쓰기를 검증하지 않으며 시각적 들여쓰기는 Forge가 자체 룰로 일괄 적용한다.
- 한 글머리 단락은 한 문장으로 작성한다.
- 소제목은 사용 여부 자유 — `1.` 섹션 직후 바로 `□`로 가도 됨. 소제목이 필요한 섹션에만 사용.

### 빈 줄

- **섹션 헤더(레벨 1) 앞에는 빈 줄 1개 이상이 필수**다.
- 소제목(레벨 2) 앞 빈 줄은 자유.
- 그 외 단락 사이 빈 줄은 작성자 자유.

```
1. 현황
가. 개요
□ (배경) 금융위는 ... 검토를 요구
 ○ 금융결제원은 ... 입장
  - 세부 근거 ...
   · 추가 보충 ...

나. 진행상황
□ (현재) 회신 대기 중

2. 이슈
□ (해석상이) 라이센스 해석이 상이
```

## 4. 요약단어

| 레벨 | 요약단어 | 형식 |
|:-:|:-:|---|
| 1 (섹션 헤더) | **불필요** | 섹션명 자체가 요약 역할 (`현황`, `이슈` 등) |
| 2 (소제목 `가.`) | **불필요** | 소제목명 자체가 요약 역할 (`개요`, `진행상황` 등) |
| 3 (`□`) | **필수** | `□ (요약단어) 본문` |
| 4~6 (`○` `-` `·`) | 사용하지 않음 | |

요약단어는 한 단어 또는 짧은 어구(한글 1~3어절).

```
□ (행정지도) 행정지도를 통한 직불업 요구 필요
 ○ 경로와 무관하게 서비스 제공 사업자에 직불업 요구가 타당
```

## 5. 주석

본문에 `*` 표기가 있는지 여부와 무관하게 두 종류를 사용할 수 있으며, **한 단락에 공존 가능**하다.

### 5.1 참조 주석 (`*`)

본문 어절 뒤에 `*`를 붙이고, 같은 단락 하단에 `*` 시작 라인으로 설명을 적는다.

```
○ 금융결제원은 스스로의 지위는 전금법 제정취지상 전자금융보조업자*라는 입장
* 전금법 제정 당시 결제중계시스템 운영자(금융결제원)을 전자금융보조업자로 명시
```

한 단락에 참조가 여러 개일 경우 `**`, `***`로 구분한다.

### 5.2 일반 주석 (`※`)

라인 시작 `※`로 시작하는 보충 설명. `*` 참조 유무와 무관하게 사용 가능.

```
□ (행정지도) 행정지도를 통한 직불업 요구 필요
※ 본 검토는 직불업 등록 의무 범위에 한함. 절차는 별도 검토.
```

## 6. 강조 (Bold)

핵심 단어를 굵게 표시할 때 **언더스코어 쌍 `__X__`** 로 감싼다.

```
□ (행정지도) __행정지도__를 통한 직불업 요구 필요
 ○ __경로 무관__으로 서비스 제공 사업자에 직불업 요구
```

### 규칙

- **Bold 표기**: `__X__` (언더스코어 두 개로 감싸기)
- **이탤릭은 사용하지 않는다.** `*X*`, `_X_` 등 단일 문자 emphasis 표기 금지.
- **markdown 표준 `**X**` (asterisk bold)는 사용하지 않는다.** asterisk(`*`/`**`/`***`)는 본 사양에서 100% 참조 주석 마커(§ 5.1)로 예약되어 있다.
- **조사·어미는 Bold에 포함하지 않는다**: 명사·어근만 감싸고 조사는 바깥에 둔다.
  - ✅ `__행정지도__를 통한` (조사 `를` 바깥)
  - ❌ `__행정지도를__ 통한`

### 작성 주체

- Sentinel STAGE 2: LLM이 핵심 명사·어근을 식별하여 자동 표기
- 손글 작성자(Forge 단독 사용 시): 직접 `__X__` 표기 가능

## 7. 결론 화살표

특정 단락이 앞선 분석의 **결론·요약**임을 표시할 때 라인 시작에 `=>`를 붙인다.

```
□ (전제) 라이센스 해석이 상이
 ○ 금융위는 직불수단으로 해석
 ○ 우리원은 일관성 부족 우려
=> 행정지도를 통해 일관된 해석 필요
```

- **표기는 `=>`로 통일**한다. `==>`, `===>>` 등 다른 길이는 본 사양에서 사용하지 않는다.
- **위치 자유** — 어느 레벨 어디에 등장해도 무방. Forge가 별도의 시각 박스(별도 마커·강조 영역)로 일괄 처리한다.
- 본문 중간(어절 뒤)에 등장하는 `=>`는 결론 마커가 아닌 일반 텍스트로 취급한다 (라인 시작에서만 마커).

## 8. Callout 박스 (`[참고]`, `[붙임]`)

본문 흐름과 별개로 **별도의 시각 박스**로 분리되어야 할 내용을 표기한다. 두 종류:

### 8.1 참고 (`[참고]`)

본문 중간에 끼어들어가는 **참조성 보충 정보**. tool2의 "참고 박스" (파란 헤더 + 흰 글씨)에 매핑.

```
[참고]
본문에 직접 들어가지 않는 보충 정보
여러 줄 작성 가능
```

- 라인 시작 `[참고]` 단독으로 박스 시작.
- 다음 줄부터 박스 내용. **빈 줄을 만나면 박스 종료**.
- 한 문서에 여러 번 등장 가능.

### 8.2 붙임 (`[붙임]`, `[붙임 N]`)

본문 끝에 첨부되는 **별도 페이지 분량의 부속 자료**. tool2의 "별지 첨부" 개념에 매핑.

```
[붙임]
첨부 본문
여러 단락 가능

[붙임 2]
두 번째 첨부
```

- 라인 시작 `[붙임]` 또는 `[붙임 N]` 단독으로 시작 (N은 정수).
- 다음 빈 줄 또는 다음 `[붙임 ...]` 마커까지가 박스 본문.
- **자동 페이지 break** — Forge가 새 페이지로 분리 후 렌더링.
- 본문 분량 초과로 Forge가 자동 [붙임] 분리할 때도 동일 형식 사용 (사용자가 직접 표기한 것과 자동 표기 동일).

### 식별 규칙

- `[참고]` `[붙임]` `[붙임 N]` 모두 **라인 시작에서 단독으로** 사용 (앞에 공백 허용, 뒤에 텍스트 금지).
- 본문 중간 `[ABC]` 등은 callout 마커로 간주하지 않음 — 일반 텍스트.
- callout 박스 내부에는 다른 글머리 (`□` `○` `-` `·`)·강조 (`__X__`)·참조(`*`) 사용 가능. 다만 **다른 callout 또는 섹션 헤더는 박스 내부에 등장 불가**.

 ○ 금융결제원은 스스로의 지위는 전금법 제정취지상 __전자금융보조업자__*라는 입장
* 전금법 제정 당시 결제중계시스템 운영자(금융결제원)을 전자금융보조업자로 명시

관련 규정: 전자금융감독규정 §13

2. 이슈
□ (해석상이) 오픈뱅킹·펌뱅킹 등 계좌이체방식 결제서비스 제공 주체에 대한 __라이센스 해석__이 상이

 ○ 경로와 무관하게 서비스 제공 사업자에 직불업 요구가 타당
※ 본 검토는 직불업 등록 의무 범위에 한함

4. 향후계획
□ (즉시) 금융위 앞 본 검토방향 의견요청

□ (장기) __수단이 아닌 행위 기반__ 라이센스체계 정비방안 건의
